# 论文写作一键流水线（paper-workflow）

把一段草稿（或一个论文文件）自动跑完 翻译/写作 → 润色 → 去 AI 味 → 逻辑检查 → 真实性核查 → 审稿，每个关键阶段暂停确认，最后汇总成给人审阅的三块结果。各阶段的具体规则由对应子 skill 定义，本 skill 负责**编排、暂停、汇总**。

> 单步需求请直接用子 skill：`/paper-translate`、`/paper-polish`、`/paper-dehumanize`、`/paper-logic-check`、`/paper-verify`、`/paper-review`、`/paper-resize`、`/paper-analysis`、`/paper-caption`、`/paper-figure`。本总控用于一次性走完主链路。

---

## 关键执行约定（务必遵守）

### 约定一：每阶段执行前，先读取对应子 skill 文件加载其完整规则

本总控**只描述编排顺序，不内嵌各阶段的 prompt 约束**。子 skill 的 prompt 原文不会随本文件自动进入上下文，因此一键全流程时**必须在进入每个阶段前，用 Read 工具读取对应子 skill 的 `SKILL.md`，逐字按其内置 prompt 与约束执行**。否则会丢失关键硬约束（LaTeX 转义、时态统一、禁列表、所有格 of 结构、宁缺毋滥等），导致质量打折。

各阶段对应文件（与本文件位于同一 `.claude/skills/` 目录下的兄弟目录）：

| 阶段 | 读取文件 |
|------|----------|
| 阶段0 翻译/写作 | `../paper-translate/SKILL.md` |
| 阶段1 润色 | `../paper-polish/SKILL.md` |
| 阶段2 去 AI 味 | `../paper-dehumanize/SKILL.md` |
| 阶段3 逻辑检查 | `../paper-logic-check/SKILL.md` |
| 阶段4 真实性核查 | `../paper-verify/SKILL.md` |
| 阶段5 审稿 | `../paper-review/SKILL.md` |

> 路径以本 `paper-workflow/SKILL.md` 所在目录为基准；若相对路径读取失败，则在 `.claude/skills/` 下按目录名定位对应 `SKILL.md`。**先读后做，不要凭记忆复述子 skill 的约束。**

### 约定二：长文档分段处理

这些 prompt 是为**段落级**文本设计的。当输入超过约 8–10 段（或明显是整章/整篇）时，按下面策略处理，不要一次性囫囵跑完：

1. **切分**：按论文结构（Abstract / Introduction / Method / Experiments / …）或自然段把全文切成「处理单元」，每个单元控制在数段以内。保留单元的原始顺序与编号。
2. **逐单元过流水线**：每个阶段（润色 / 去AI味）对**每个单元分别**套用子 skill 规则，保持各单元内部的转义、时态、术语一致。
3. **跨单元一致性**：切分时记录全文统一的术语表与缩写（如 LLM 不展开），各单元处理时沿用，避免同一概念在不同章节被改成不同名字。
4. **审稿与核查阶段**：阶段3 逻辑检查、阶段4 真实性核查、阶段5 `paper-review` 均针对**整篇拼合后的文本**做评估（看的是全局一致性、事实正确性与贡献，不能分段审）。
5. **暂停粒度**：默认仍以「阶段」为暂停点（一个阶段把所有单元处理完后统一暂停展示）；若单元很多导致单轮输出过长，可在阶段内按单元分批展示，但阶段末仍统一确认一次再进下一阶段。

### 约定三：支持文件模式（给路径，而非只贴正文）

用户可以**贴正文**，也可以**给文件路径**（如 `paper/main.tex`、`sections/method.tex`、投稿 `.docx`）。给路径时：

1. **Read 读入**：用 Read 工具读取目标文件作为流水线输入；若用户指定了某一节（如"只处理 Method"），在文件中定位对应 `\section`/章节范围。
2. **就地回写（需确认）**：各阶段产出确认后，可用 Edit 把结果**写回原文件对应位置**，而不是只在对话里贴一份让用户手动粘贴。**首次回写前必须明确征得用户同意**（"是否直接改写 main.tex？"），并建议用户先 commit 或备份。
3. **保边界**：回写时严格保留改动段落之外的内容、LaTeX 导言区、`\label`、图表环境等，不要顺手"整理"未要求改动的部分。
4. **数据/引用旁路**：若同目录有 `.bib`、`results/`、实验数据表，记下路径，供阶段「真实性核查」读取。
5. **降级**：无文件读写权限或路径不可达时，自动退回"贴正文→吐文本"模式，并告知用户。

> 文件模式是本工作流相对"手动复制 prompt"的关键增量：让 agent 直接在用户的论文文件上工作。但**回写是有副作用的操作，务必先确认、保备份、保边界**。

## 阶段 0：输入检测（不暂停，直接判定后告知用户）

检测三件事，并向用户播报判定结果：

1. **格式**：文本含 `\cite{}`/`\ref{}`/`\begin{...}`/`$...$`/`\section` 等 → 判为 **LaTeX**；否则判为 **Word / 纯文本**。
2. **语言**：按中英文字符占比判主语言。
3. **篇幅**：估算段落数。超过约 8–10 段或为整章/整篇时，按「约定二：长文档分段处理」先切分为处理单元，并在播报中说明切分方案。

据此决定起点（**需要翻译时，先 Read `../paper-translate/SKILL.md` 再按其规则执行**）：

- **中文草稿** → 先做翻译/重写（按 `paper-translate` 的内置 prompt）：
  - LaTeX 场景 → 中转英-latex（模式 A）
  - Word 场景 → 默认中转英-word（模式 C）；若用户明确要保持中文论文，则用中转中-word（模式 D）
  - 目标语言不明时，用一句话问用户"产出英文还是中文论文"，其余按格式自动定。
  - 保留译文的 **Part 2 中文直译**，供最终对照复用。
- **英文初稿** → 跳过翻译，直接进入阶段 1。

播报示例："检测到：中文草稿 + LaTeX 格式 → 将先中转英，再依次润色、去AI味、审稿。"

---

## 阶段 1：润色（结束后暂停）

**先 Read `../paper-polish/SKILL.md`**，对当前英文文本逐字套用其规则（英文 LaTeX 走深度润色；若产出为中文论文则走中文克制润色）。长文档时对每个处理单元分别润色。
产出 Part 1 润色文 + Part 3 修改日志。

**暂停**：展示润色文与修改日志，提示——
"【阶段1 润色完成】确认进入下一阶段（去 AI 味）请回复继续；需要调整请直接说明。"
等用户回应后再继续。用户的微调要求在本阶段就地处理，满意后再进入阶段 2。

---

## 阶段 2：去 AI 味（结束后暂停）

**先 Read `../paper-dehumanize/SKILL.md`**，对阶段 1 的产出逐字套用其规则（英文 LaTeX / 中文 Word 自动分流，宁缺毋滥）。长文档时对每个处理单元分别处理。
产出 Part 1 文本 + 修改日志（或"检测通过"评价）。

**暂停**：展示结果，提示——
"【阶段2 去AI味完成】确认进入下一阶段（逻辑检查）请回复继续；需要调整请直接说明。"

---

## 阶段 3：逻辑检查（结束后暂停）

**先 Read `../paper-logic-check/SKILL.md`**，对阶段 2 的产出逐字套用其红线审查规则。这是正文最后一次自动改动后的安全网：高容忍度，只揪致命错误（前后矛盾、术语不一致、严重 Chinglish/语病），不做风格优化。长文档时**针对整篇拼合后的文本**核对，重点查跨章节的术语一致性与逻辑矛盾。
产出：若无致命问题，输出"[检测通过，无实质性问题]"；否则中文分点列出必须修改项。

**暂停**：展示检查结果，提示——
- 若检测通过："【阶段3 逻辑检查完成】未发现致命问题。确认进入下一阶段（真实性核查）请回复继续。"
- 若发现问题："【阶段3 逻辑检查完成】发现以下需修正的问题：……。建议先修正再进入真实性核查；确认如何处理。"

被标出的致命问题应就地修正（或按用户指示处理）后再进入下一阶段，并把这些问题汇入最终汇总的「修改意见」。

---

## 阶段 4：真实性核查（结束后暂停）

**先 Read `../paper-verify/SKILL.md`**，对阶段 3 核对后的产出逐字套用其规则。逻辑检查管"读得通不通"，本阶段管"是不是真的"：核对引用真实性、正文数字与实验表一致性、有无 overclaim。**若有 `.bib` / 实验数据文件，先 Read 读入以做外部核对**；否则只能做内部自洽并声明覆盖范围。长文档时针对整篇拼合后的文本核查。
产出：`[OK]` / `[CITATION NEEDED]` / `[MISMATCH]` / `[NUMBER MISMATCH]` / `[OVERCLAIM]` 分类的核查报告。

**关键**：本阶段**只标记、不编造**——核不准的引用绝不替用户补全，对不上的数字只标出供人裁定。

**暂停**：展示核查报告，提示——
- 若全部通过："【阶段4 真实性核查完成】未发现明显事实硬伤（标红项仍建议人工终核）。确认进入下一阶段（审稿）请回复继续。"
- 若有标记项："【阶段4 真实性核查完成】发现引用/数字/宣称层面的待处理项：……。这些属于不能自动改写消除的事实性风险，确认如何处理。"

标出的 `[CITATION NEEDED]` / `[NUMBER MISMATCH]` / `[OVERCLAIM]` 全部汇入最终汇总的「修改意见」，并标为**必须人工处理**的高优先级项。

---

## 阶段 5：审稿（结束后暂停）

**先 Read `../paper-review/SKILL.md`**，对阶段 4 核查后的产出逐字套用其规则（无 PDF 时审当前文本）。长文档时**针对整篇拼合后的文本**做评估，不分段审。先确认投稿目标（如 ICML 2026）。
产出 Part 1 审稿报告（Summary/Strengths/Weaknesses/Rating）+ Part 2 Strategic Advice。

**暂停**：展示审稿报告，提示——
"【阶段5 审稿完成】确认生成最终汇总请回复继续；需要先按审稿意见改稿请说明。"

---

## 阶段 6：最终汇总输出（给人审阅）

将全流程结果整理为以下三块，一次性输出：

### 一、修改方向
来自审稿 Part 2 Strategic Advice：问题根源、可救性判断（哪些修订期可解决、哪些是方法层结构性缺陷）、行动指南（补什么实验、重写哪段、Rebuttal 如何降攻击面）。

### 二、修改意见
逐条汇总：**阶段4 真实性核查标出的引用/数字/宣称问题（最高优先级，必须人工处理）** + 审稿 Part 1 的 Weaknesses（具体到实验设置/论证/表述）+ 阶段3逻辑检查标出的致命问题 + 阶段1润色与阶段2去AI味的修改日志要点。每条尽量可操作。

### 三、中英文完全对照
最终英文稿与其中文直译**逐段并排**呈现，便于人工核对逻辑是否符合原意。中文直译复用阶段0/各阶段产出的 Part 2，必要时对最终定稿补做一次直译以保证段段对应。格式示例：

```
[段落 1]
EN: <最终英文段落>
ZH: <对应中文直译>

[段落 2]
EN: ...
ZH: ...
```

## 编排注意事项

- **先读后做**：每阶段进入前必须 Read 对应子 skill 文件并按其原文执行，不要凭记忆复述约束（见「约定一」）。
- **长文档分段**：超长输入按处理单元逐段过流水线，审稿阶段对整篇评估（见「约定二」）。
- **关键阶段必须暂停**（阶段1/2/3/4/5 后），不要一口气跑到底；这是用户明确要求的可控点。
- 阶段间传递的是上一阶段 Part 1 的正文，修改日志/逻辑检查问题/真实性核查标记/审稿意见单独留存供最终汇总。
- 全程保持原 prompt 的硬约束（LaTeX 转义、时态、禁列表、禁多余加粗、所有格 of 结构、宁缺毋滥等），不要在编排层放松这些规则。
- 若用户在某阶段要求"只跑到这里"，则止于该阶段并直接给出当前产物。
